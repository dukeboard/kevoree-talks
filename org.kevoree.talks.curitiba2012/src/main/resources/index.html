What is Model@Runtime ?
  • Abstraction layer
    • Architectural model which defines how the system configured
      • Nodes
      • Components and their interactions
So what is Model@Runtime ?
First of all, it's an abstraction layer which define the architecture of the system:
For instance, the nodes, the components that are executed on theses nodes and the interaction between these components
What is Model@Runtime ?
  • Abstraction layer
  • A reflexion layer
    • DAS are build as a reflexive (introspection and intercession) systems
      • 2 Strongly connected intercession layer with the running system
        • Misconfiguration can crash the system
    • Offline reflexion
      • Test and validate configuration before applying it
Standard reflexion layers such the one available in Java are strongly connected to the runtime and so changes ont the configuration at the reflexion layer is directly applied on the runtime And this can lead to misconfigurations that crash the system
So With Model@Runtime, the reflexion layer is an offline reflexion layer that means, changes on the reflexion layer are not directly applied on the system and we are able to test and validate it before.

A model abstract nodes configurations

Here, it's the model which define the architecture of the system with some nodes, some components

Each node host a model@runtime engine

Ñ Ñ Ñ
Each node host a M@R engine

Model@Runtime process

And this engine is able to reconfigure the system according to model reconfigurations So First, a new model is proposed This model is verified If it is ok, this new model is compared with the current one to find all the needed commands to update the system These commands are applied and if all is ok, the new configuration is saved as the current one

Kevoree
A Model@Runtime implementation for Distributed Dynamically Adaptive systems

So this is how the Model@Runtime works and we propose Kevoree which implements this process
  • A Component-based approach
    • ComponentType
      • Encapsulate business features
  • Extend with distributed features
    • ChannelType
      • Encapsulate communication semantics
    • NodeType
      • Encapsulate runtime heterogeneity
      • Encapsulate adaptation semantics
    • GroupType
      • Encapsulate model@runtime dissemination semantics
So kevoree provide a Model@Runtime implementation
with components
and with Channels
And this Component-based approach is extend with some distributed features
There are the nodes
There are also the groups

KevScript
A language to build adaptive system

Kevoree Script (KevScript) language
to build reasonning engine

        merge "mvn:org.kevoree.corelibrary.javase/org.kevoree.library.javase.javasenode"
        updateDictionary instance1 {port="8080"}
        addComponent component1@node1:ComponentType1
        addChannel channel1 : ChannelType1
        bind component1.port1@node1 -> channel1
        unbind component3.port1@node1 -> channel1
        removeComponent component3@node1
        removeChannel channel2
        moveComponent component1@node1 -> node2
        addGroup group1 : GroupType1
        removeGroup group1
        addToGroup node1@group1
        removeFromGroup node1@group1
        addNode node1 : NodeType1
        removeNode node1
        network node1 -> node2 {"ip" = "192.168.0.1"}
        addNode node1 : NodeType1
        removeNode node1
        addChild node2@node1
        removeChild node2@node1
Reasoning engines are able to propose new configuration for the system by building a new model that they submit on the Model@Runtime process
The KevScript language ease the way new model can be built
With these commands, a reasoning engine can easily add a new component, bind it with another one, etc

Cloud Systems
by Erwan Daubert, François Fouquet, Triskell Team INRIA

Managing cloud layers with Kevoree node

  • Kevoree node is a container, of components but also of nodes !
  • A parent node has responsibility for child's nodes life cycle (start/stop)
  • NodeType refine adaptation strategy
  • Think the Cloud as a hierarchy

IaaS and PaaS nodes

  • Specific adaptation primitives: Add/Remove node
  • Available NodeType
    • MiniCloud : Add/Remove Java Virtual Machine
    • FreeBSD Jails: Add/Remove VM in the same kernel space
    • Amazon EC2: Add/Remove EC2 Virtual Machine (on going work)
    • LibVirt: Add/Remove VM with KVM/XEN/Virtualbox (on going work)
