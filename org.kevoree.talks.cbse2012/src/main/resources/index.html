Challenges of adaptation on μC

μC node can't manage every adaptation capability !
Dynamic adaptation on μC face of several challenges
  • Downtime must be minimal
  • Volatile memory usage must fit in few kB
  • Limit persistent memory write
  • Need stateful reconfiguration, ability to recover power failure

>μKevoree

μKev Component model

  • Type definition / instance (what / where)
  • Component with ports (in/out) and parameters
  • Communication channels between ports (bindings)
  • Write code (C/Processing/Java) 
    @ComponentType
@DictionaryType({
        @DictionaryAttribute(name = "pin")
})
@Requires({
        @RequiredPort(name = "temp")
})
public class TempSensor
    extends AbstractPeriodicArduinoComponent {
    //behavior code
}
  • Generate firmware for types + KevScript for instances

Behind the scene

  • Asynchronous message passing
  • FIFO semantic on each ports
  • No OS == no multi-threading
  • Opportunistic scheduler
    • FIFO remains size
    • Periodic executions
  • Transactional KevScript interpreter
  • Token ring dynamic memory allocation
  • Flat generated reflexive layer for dynamic instantiations

Dynamic adaptation with (μ)Kevoree

  • Level 1 : Parametric adaptation : parameter update
    • Action : send adaptation (kev)script
  • Level 2 : Architectural adaptation
    • Action : send adaptation (kev)script
  • Level 3 : Dynamic provisioning of types
    • Components, Channels <-> C Struct
    • Action : generate new firmware, reboot
  • Level 4 : Adaptation for remote management
    • Limited action : forward script to peers

Kevoree + μKevoree properties

μKevoree model@runtime interface to μC
  • Allow get a model representation of internal state
  • Allow incremental update remotely with short downtime
Using model@runtime on all CPS entities
  • Homogeneous reflexion layer on all CPS
  • Allow global reasoning and seamless adaptation
