Distribute tests execution for Continuous Integration process

Continuous Integration context

continuous integration (CI) is the practice of merging all developer workspaces with a shared mainline several times a day. ...CI was originally intended to be used in combination with automated unit tests written through the practices of Test driven development.
  • Continuous Integration is time consuming
    • Especially the test execution
    • Apache Camel: 7205 tests, 1h 23min
    • Rothermel et al. says some projecs needs more than 7 weeks

Proposed solution

  • Dynamically distribute the tests execution on Cloud
    • Dynamically building the complex application representing the tests execution
    • Dynamically building the needed platform
      • Managing the specific requirement of the tests (OS, RAM, etc)
    • Reusing already defined platform

Results

  • Decrease the tests execution time: speedup = 3.40 (around 40 minutes)
    • The much longer test is around 20 minutes
      • limit the speedup
  • Able to reuse already defined platform
    • Keep alive platform help to reduce the time needed to start VMs
  • Use lightweight virtualization
    • using user space virtualization instead of hardware virtualization if possible
      • Managed by the IaaS taking into account PaaS specificities

Distributed web server

Distributed web server context

  • Need to manage Elasticity and Scalability
    • Replication of web pages
    • Load balancing between requests
  • Dynamicaly managed by the platform

Proposed solution

  • Defining web server and web pages as Kevoree Components
  • Using specific communication channel
    • Able to do load balancing between components
  • Able to easily duplicate web pages
  • Able to easily migrate web pages
  • Monitoring of CPU load

Results

  • IaaS take into account Components interaction
    • To host platform on the same infrastructure node
  • IaaS and PaaS managers coordination
    • Avoid VM migration
    • Replaced by components migration or creation of cache
