Implémentation d'une solution d'infrastructure hybride

Différents types de Clouds

  • Clouds publics
    • Utilisable au travers d'Internet
  • Clouds privés
    • Utilisable au sein d'une entreprise et seulement sur son réseau
  • Clouds hybrides
    • Composition de plusieurs Clouds privés et/ou publics
      • Pour l'usage de ressources spécifiques
      • Pour pallier le manque de ressources au sein de l'entreprise

Protocole expérimental

  1. Définition de nouveaux types de nœuds d'infrastructure
  2. Définition d'un gestionnaire d'infrastructure
  3. Évaluation sur la quantité de code nécessaire*
    • Le code généré n'est pas pris en compte
    • Comparaison par rapport au code réutilisé

* utilisation de cloc - http://cloc.sourceforge.net/

Implémentation

  • Utilisation d'un système de virtualisation d'espace utilisateur (Jails*1)
    • Partage du système d'exploitation
    • Séparation de l'espace de stockage
    • Séparation de l'adressage réseau
  • Proxy vers un Cloud public (Amazon EC2*2)
    • virtualisation du système d'exploitation

*1http://www.freebsd.org/doc/en/books/handbook/jails.html
*2http://aws.amazon.com/en/ec2

Gestionnaire d'infrastructure

  1. Identification des nœuds de plate-forme non hébergés
  2. Identification du système d'exploitation demandé
    • Si Windows alors allocation sur le proxy Amazon EC2
    • Sinon allocation sur l'une des machines du Cloud privé
      • Répartition équitable des nœuds de plate-forme
  3. Soumission du nouveau modèle
    • Dissémination par le(s) groupe(s)
    • Mise en place des nœuds de plate-forme
      • Par les nœuds parents (qui héberge)

Protocole expériemental

  1. Soumission de modèle de plate-forme
  2. Évaluation du temps d'exécution
    • Pour l'intégration des nœuds de plate-forme sur l'infrastructure

Intégration Continue

L'intégration continue est un ensemble de pratiques utilisées en génie logiciel consistant à vérifier à chaque modification de code source que le résultat des modifications ne produit pas de régression dans l'application développée.*1
  • L'intégration Continue est consommatrice en temps
    • Notamment l'exécution des tests
    • Certains projets nécessite plus de 7 semaines*2

*1http://fr.wikipedia.org/wiki/Intégration_Continue
*2Rothermel, G. et al - Prioritizing test cases for regression testing - 2001

Solution proposée

  • Effectuer l'exécution des tests sur un Cloud
    • En parallélisant leur exécution
    • En définissant la plate-forme d'exécution nécessaire
      • En tenant compte des contraintes spécifiques à chaque test (OS, RAM, ...)

Cas d'usage: Apache Camel 2.8 (http://camel.apache.org/)

  • Implémentation de patrons d'intégration *
    • facilite l'intégration d'applications en normalisant les communications entre elles
  • 175 modules, 3875 suites de tests, 8084 tests
  • Temps d'exécution: 1h40min (sans le Cloud)
    • Intel R, AtomTM D425 1.8GHz avec 8 Go de mémoire vive
  • Temps d'exécution: 40min (sur le Cloud)
    • 10 machines
    • Intel R, AtomTM D425 1.8GHz avec 8 Go de mémoire vive

*Hohpe, G. et al. - EIP: Designing, building, and deploying messaging solutions - 2004.

Sur le Cloud:
7 min pour initialiser l'ensemble des nœuds de plate-forme 20 min pour le test le plus long

de l'adaptation multi-niveaux ?
Distribution d'un serveur web

Protocol expérimental

  • Implémentation de gestionnaires
    • Infrastructure
      • Utilisation des informations multi-niveaux
    • Plate-forme
      • Coordination de l'adpatation entre Infrastructure et Plate-forme

Implémentation

  • Définition d'un serveur web distribué
    • Définition d'un serveur qui écoute sur un port réseau
    • Définition de page web répondant pour une url spécifique
    • Interconnexion du serveur et des pages grâce aux channels
  • Implémentation d'un gestionnaire d'infrastructure
    • Tenant compte des interactions au niveau application
      • Pour le positionnement des machines virtuelles
      • Pour rapprocher les machines virtuelles ayant beaucoup d'interactions
  • Implémentation d'un gestionnaire de plate-forme
    • Capable de refuser des adaptations provenant de l'infrastructure
      • Pour éviter les migrations de VMs
      • Pour les remplacer par des adaptations au niveau plate-forme

Évaluation

  • Prise en compte des informations de plusieurs niveaux
  • Coordination de l'adpatation entre plusieurs niveaux
  • Cas d'étude spécifique
cas d'étude spécifique:
Les gestionnaires sont des implémentations spécifiques qui tiennent compte de certains informations et qui sont capable de se coordonner dans le cadre de certaines adaptations
C'est le modèle qui permet de définir ces implémentations spécifiques.
