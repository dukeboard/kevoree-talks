Adaptation & Cloud systems

IBM frequently cites four aspects of self-management ... Ultimately, these aspects will be emergent properties of a general architecture, and distinctions will blur into a more general notion of self-maintenance.
  • Each adaptation has impact on the running system
  • Each adaptation may have side effects
    • Side effects may have impact on the other adaptations
  • We need to coordinate adaptation between layers

How to coordinate adaptation ?

  • Use only one adaptation engine that manage all layers
    • Layers must be independent (e.g: different providers)
  • Coordinate all adaptation engines
  • We need to have a common abstraction for all the adaptation engines
  • We need to represent all layers together

Kevoree
A Model@Runtime implementation for Distributed Dynamically Adaptive systems
François Fouquet 's thesis

What is Model@Runtime ?

  • Abstraction layer
    • Architectural model which defines how the system is configured
      • Components and their interactions
      • Nodes
So what is Model@Runtime ?
First of all, it's an abstraction layer which define the architecture of the system:
For instance, the components and the interaction between these components but also the nodes that hosts these components

What is Model@Runtime ?

  • Abstraction layer
  • Offline reflexion layer
    • Usually DAS are build as a online reflexive systems
      • Â Strongly connected with the running system
        • Â Misconfiguration can crash the system
    • Test and validate configuration before applying it
Model@Runtime is also a reflexion layer. Standard reflexion layers such as the one available in Java are strongly connected to the runtime and so changes ont the configuration at the reflexion layer is directly applied on the runtime And this can lead to misconfigurations that crash the system u So With Model@Runtime, the reflexion layer is an offline reflexion layer that means, changes on the reflexion layer are not directly applied on the system and we are able to test and validate it before.

Model@Runtime process

And this engine is able to reconfigure the system according to model reconfigurations So First, a new model is proposed This model is verified If it is ok, this new model is compared with the current one to find all the needed commands to update the system These commands are applied and if all is ok, the new configuration is saved as the current one

How Kevoree modelize Cloud systems ?

  • Components

    • Layer management (PaaS, IaaS)
  • Nodes
    • Software runtime (SaaS)
    • VM system (PaaS)

  • But we need to modelize all the layers together

Distribute tests execution for Continuous Integration process

Continuous Integration context

>continuous integration (CI) is the practice of merging all developer workspaces with a shared mainline several times a day. ...CI was originally intended to be used in combination with automated unit tests written through the practices of Test driven development.
  • Continuous Integration is time consuming
    • Especially the test execution
    • Apache Camel: 7205 tests, 1h 23min
    • Rothermel et al. says some projecs needs more than 7 weeks

Proposed solution

  • Dynamically distribute the tests execution on Cloud
    • Dynamically building the complex application representing the tests execution
    • Dynamically building the needed platform
      • Managing the specific requirement of the tests (OS, RAM, etc)
    • Reusing already defined platform

Results

  • Decrease the tests execution time: speedup = 3.40 (around 40 minutes)
    • The much longer test is around 20 minutes
      • limit the speedup
  • Able to reuse already defined platform
    • Keep alive platform help to reduce the time needed to start VMs
  • Use lightweight virtualization
    • using user space virtualization instead of hardware virtualization if possible
      • Managed by the IaaS taking into account PaaS specificities

Distributed web server

Distributed web server context

  • Need to manage Elasticity and Scalability
    • Replication of web pages
    • Load balancing between requests
  • Dynamicaly managed by the platform

Proposed solution

  • Defining web server and web pages as Kevoree Components
  • Using specific communication channel
    • Able to do load balancing between components
  • Able to easily duplicate web pages
  • Able to easily migrate web pages
  • Monitoring of CPU load

Results

  • IaaS take into account Components interaction
    • To host platform on the same infrastructure node
  • IaaS and PaaS managers coordination
    • Avoid VM migration
    • Replaced by components migration or creation of cache
