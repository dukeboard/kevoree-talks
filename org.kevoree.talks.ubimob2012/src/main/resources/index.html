Heterogeneous dissemination strategy

Distributed system usually organized in clusters (group of nodes) Considering a cluster of 3 nodes p sharing a configuration model :
  • Sporadic context : N1 can update N2 without synchronizing N3
    N1 N2 N3
  • Cluster context : N1 must synchronize N2 & N3 in the same logical time
    N2 N1 N3
  • Opportunistic nodes : A can synchronize B or C depend of previous configurations
    N2 N1 N3

Mix them all !

Complex architectures, especially on field sporadic usage, lead to composition of dissemination strategies
  • M1 & M2 are mobile nodes, group will use partial order
  • S1 & S2 are cluster nodes, group will use total order
  • A partial order group is between M2 & S2.
  • M1 S1 3 S2 3 M2
  • mess ... 2

Dissemination scope

Global synchronization is not an option for ubiquitous architecture
Need different scope of this synchronization defined by the group Considering two group of nodes: (A,B) & (B,C)
  • A & B share context, B & C share another context
  • A & C don't share anything, cluster defines synchronized elements
  • A & C can be desynchronized if cluster 2 ensures a partial order
  • mess ... 2

Architectural abstraction needed !

Claim :
To address all these problems, we need to provide an abstraction to manipulate and compose dissemination strategies

>Kevoree Project

Model@Runtime for DDAS

  • MDE@Runtime
    • Shared model representation for distributed nodes
    • Offline & online operation, compute@Model level, apply @Runtime
  • Component-based
    • Communication semantics between component in channel
  • Heterogeneity management with NodeType
    • Java Node, Dalvik Node, Arduino Node
    • Cloud Node (Jails/*BSD, JCloud, mini-cloud, EC2)

Kevoree concepts

  • ComponentType
    • Encapsulate domain features
  • ChannelType
    • Encapsulate communication semantics
  • GroupType
    • Encapsulate model@runtime dissemination semantics
  • NodeType
    • Encapsulate adaptation semantics

>Daum Project

Daum challenges

Define a tactical on-field operating system for firefighters

    Complex adaptative ubiquitous system because:
  • Adaptation inherent to domain
    firefighters are deployed also dynamically
  • Heterogeneous (sensors , embedded computer, tablet )
  • Complex topologies and nodes distributions
    • sporadic
    • several communication possibilities

Daum in a nutshell

Embedded sensors on firefighters personal equipment
monitor temperature, heart rate 4
Deploy embedded tablets to collect tactical and sensors data.
Deploy embedded computers that interact with central system in cars.

  • Sensors are low consumption µC nodes
  • Cars and tablets are middle class consumption computer (ARM9-11)
  • Central system is a legacy grid deploy system
